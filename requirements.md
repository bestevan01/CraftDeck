# CraftDeck 요구사항 정의서

## 1. 개요

라즈베리파이에 apt 패키지 형태로 설치하여 백그라운드 서비스로 상시 구동하고, 같은 네트워크(LAN)의 다른 기기에서 웹 브라우저로 접속해 **마인크래프트 자바 에디션(Java Edition)** 서버를 손쉽게 구성·관리할 수 있는 프로그램. 옵션으로 공유기 포트 포워딩을 설정하여 외부(인터넷)에서도 접속·관리할 수 있는 기능을 제공한다. 베드락 에디션은 지원 대상이 아니다.

## 2. 목표

- apt 패키지 설치만으로 별도 런타임 설치 없이 즉시 사용 가능해야 한다.
- systemd 서비스로 백그라운드 상시 실행되어야 한다.
- LAN 내 어떤 기기에서든 웹 브라우저로 접속해 관리 UI를 사용할 수 있어야 한다.
- 여러 개의 마인크래프트 서버 인스턴스를 동시에 생성·설정·실행·중지·삭제할 수 있어야 한다.
- 서버 생성 시 구동기(Loader)를 목록에서 선택하거나 직접 업로드할 수 있어야 한다.
- 플러그인/모드를 목록에서 선택하거나 직접 업로드하여 서버에 적용할 수 있어야 한다.
- 웹에서 실시간 콘솔(로그 확인 + 명령어 입력)을 사용할 수 있어야 하며, 동시에 자주 쓰는 명령을 GUI 버튼으로도 조작할 수 있어야 한다.
- 필요 시 공유기 포트 포워딩을 앱에서 직접 설정하여 외부 네트워크에서도 접속·관리가 가능해야 한다.

## 3. 범위

### 3.1 포함
- 마인크래프트(자바 에디션) 서버 인스턴스 생성/설정/삭제 (버전, 포트, 최대 플레이어, JVM 옵션, 자원 제한)
- 구동기(Loader) 선택 설치(공식 배포처 자동 다운로드) 또는 커스텀 jar 직접 업로드
- 플러그인/모드 선택 설치 또는 직접 업로드, 개별 활성화/비활성화/삭제
- 서버 시작/중지/재시작
- 실시간 콘솔 로그 스트리밍 및 명령어 입력(RCON 기반) — 항상 사용 가능한 기본 옵션
- 자주 쓰는 명령의 GUI 버튼 조작 (실시간 콘솔과 병행 제공, 콘솔을 대체하지 않음)
- 서버별 CPU/메모리 자원 제한
- 월드/설정 파일 백업 및 복원
- 사용자 인증(로그인) 및 세션 관리
- 공유기 포트 포워딩 자동/수동 설정 (UPnP/NAT-PMP, 실패 시 수동 안내)
- 외부 접속을 위한 동적 DNS(DDNS) 연동 옵션
- 외부 노출 시 강화된 보안 옵션(HTTPS, 2단계 인증 등)

### 3.2 제외 (1차 범위 밖)
- 베드락 에디션 지원
- 멀티 라즈베리파이(클러스터) 간 서버 분산 관리
- 결제/과금 기능

## 4. 기능 요구사항

### 4.1 구동기(Loader) 설치

- FR-1: 1차 출시에서 지원하는 구동기는 **Vanilla, Paper, Purpur, Folia, Pufferfish, Leaf, Fabric, NeoForge, Velocity**이다 (Spigot/Quilt 등은 이후 확장 대상). Vanilla는 Mojang 공식 `version_manifest.json` API에서 서버 jar를 그대로 받아오면 되어 구현 난이도가 가장 낮다.
- FR-1e: **(결정 반영)** 최초 목표에 있던 **오리지널 Forge는 지원 목록에서 제외**했다 — Forge는 버전마다 자주 바뀌는 벤더 전용 설치 스크립트를 거쳐야 해서 안정적인 자동화가 어려운 반면, 사실상의 후속 프로젝트인 **NeoForge**는 Maven 버전 목록 + 인스톨러 + 공식 `ServerStarterJar` 조합으로 다른 로더와 동일하게 `java -jar server.jar`로 실행할 수 있는 형태를 재현할 수 있어 NeoForge로 대체했다. 또한 애초 계획에 없었던 **Purpur, Folia, Pufferfish, Leaf**도 각각 고유의 안정적인 배포 API가 존재해 구현 난이도가 낮았으므로 1차 목록에 함께 추가했다.
- FR-1a: **(결정 반영)** 프록시는 **Velocity만 지원**한다. BungeeCord는 사실상 유지보수가 멈춘 레거시 프로젝트이고, 그 포크인 Waterfall조차 PaperMC 팀이 "Velocity를 쓰라"며 유지보수를 중단한 상태라 1차 범위에서 제외한다. 프록시(Velocity)는 일반 게임 서버(월드를 직접 구동)가 아니라 **여러 백엔드 마인크래프트 서버를 하나의 포트에서 라우팅하는 역할**이며, 앱은 이를 별도의 인스턴스 유형(프록시)으로 취급한다.
- FR-1b: **(결정 반영)** 프록시는 운영자가 직접 생성하지 않는다 — 프록시 뒤에 둘 첫 서버가 만들어지는 시점에 CraftDeck이 싱글턴 Velocity 프록시를 필요하면 자동으로 생성한다(`ensureProxyInstance`). Paper 계열 및 포워딩 모드가 설치된 Fabric/NeoForge 서버는 생성 시 기본적으로 이 프록시의 백엔드 목록에 자동 등록되며(독립 노출을 명시적으로 선택하지 않는 한), **생성된 순서가 곧 우선순위**가 되어 먼저 만든 서버가 1순위(기본 접속 서버)가 된다. 이렇게 자동 관리되는 목록을 운영자가 직접 재배치(우선순위 변경, forced-host 지정, 특정 서버 제외)할 수 있는 API(`PUT /api/instances/{id}/proxy/backends`)도 이미 존재하지만, 아직 전용 UI는 없고 API로만 가능하다. **(FR-1f로 갱신)** 이 "기본적으로 프록시 뒤에 등록" 동작은 이제 소유 도메인이 실제로 연결되어 있을 때만 적용된다 — 아래 FR-1f 참고.
- FR-1c: Velocity 프록시의 경우, 서브도메인별로 서로 다른 백엔드 서버로 라우팅하는 "forced host" 설정을 지원하여, 향후 메인 도메인+프록시 조합 환경에서 서버별 SRV 레코드 없이도 하나의 포트로 서브도메인 기반 라우팅이 가능하도록 한다 (FR-29의 SRV 방식과 별개로, 프록시를 쓰는 경우의 대안 경로).
- FR-1d: **(결정 반영)** 별도의 헬스 체크 폴러를 CraftDeck이 직접 구현하는 대신, Velocity 자체가 이미 제공하는 두 메커니즘을 그대로 활용한다: (a) `try`/`forced-hosts` 목록은 접속 시도마다 매번 새로 순서대로 시도되므로, 1순위 서버가 응답하지 않으면 자동으로 2순위로 넘어가고 — 1순위가 복구된 뒤의 **새 접속(또는 재접속)**부터는 다시 자동으로 1순위가 우선된다(같은 서브도메인을 여러 서버에 지정하면 그 서브도메인 전용 우선순위 목록이 되도록 `writeVelocityConfig`에서 그룹핑 처리, `internal/api/handlers_proxy.go`), (b) 이미 접속 중인 플레이어가 있는 서버가 갑자기 끊기면 `failover-on-unexpected-server-disconnect = true` 설정으로 다음 순위 서버로 자동 이동한다. **범위에서 제외한 것**: 이미 폴백 서버에 접속해서 플레이 중인 기존 세션을 1순위 서버가 복구되자마자 강제로 다시 옮기는 능동적 마이그레이션(별도의 프록시 플러그인이 필요하며, 강제 재접속으로 인한 체감 끊김이 있어 이번 범위에서 제외), 그리고 이 자동 원복 동작 자체를 끄는 수동 전환 설정(활용하는 메커니즘이 전부 Velocity의 "접속 시도마다 재평가" 방식이라 끌 수 있는 별도 스위치가 없음).
- FR-1f: **(결정 반영/구현 완료)** Velocity 프록시는 **소유한 메인 도메인이 실제로 연결되어 있을 때만** 기본 프록시로 동작한다 — forced-host 서브도메인 라우팅(FR-1c)은 실제로 그 이름을 풀어줄 DNS가 있어야 의미가 있는데, 도메인이 아예 없거나 무료 DDNS 서브도메인(FR-26의 `free_subdomain`, 정의상 서버 1개에만 배정 가능 — FR-27)만 있는 경우엔 서브도메인 기반 접속 자체가 애초에 불가능하기 때문이다. 이 경우 CraftDeck은 **Velocity를 아예 비활성화**하고, 여러 서버를 포트 번호로 직접 구분해 접속하는 방식(각 서버를 독립 노출로 전환, 서버마다 다른 `game_port`)으로 자동 전환한다. 판단 기준은 `internal/ddns`(도메인 등록 여부/방식 저장 — FR-26~31의 실제 DNS API 연동/SRV 자동 갱신/소유권 검증도 이후 단계에서 모두 구현 완료됨)의 등록 상태이며, `main_domain` 등록 여부가 바뀔 때마다(등록/해제) `ReconcileProxyMode`가 즉시 실행되어: (a) 메인 도메인이 새로 연결되면 독립 노출 중이던 프록시 지원 가능 서버들을 자동으로 프록시 뒤에 재등록하고, (b) 메인 도메인이 해제되면 프록시에 등록된 서버들을 모두 독립 노출로 자동 전환한 뒤 프록시 인스턴스 자체를 완전히 삭제한다(고정 1GB 메모리를 낭비하지 않도록). 데몬 재시작 시에도 한 번 더 반영되어 드리프트를 방지한다.
- FR-1g: **(실기 검증 결과 반영)** Velocity는 바닐라 마인크래프트 프로토콜 구조를 엄격히 디코딩할 수 있는 패킷만 통과시킬 수 있다 — 커스텀 플러그인/페이로드 채널(임의 바이트, 대부분의 Fabric/NeoForge 모드가 쓰는 방식)은 그대로 통과되지만, **엔티티/블록 상태 등 바닐라 패킷 자체의 직렬화 구조를 변형하는 모드**(대표적으로 Create — 컨트랩션·좌석 엔티티 등)가 설치된 백엔드 서버는 Velocity가 그 패킷을 "디코딩 실패(`A packet did not decode successfully`)"로 판단해 접속을 강제로 끊는 현상이 실기 테스트에서 확인됐다(Velocity 로그: `com.velocitypowered.proxy.util.except.QuietRuntimeException`). Velocity/BungeeCord 계열 프록시 전반의 알려진 한계로, CraftDeck이 설정으로 고칠 수 있는 문제가 아니다. 이런 모드가 설치된 Fabric/NeoForge 서버는 **독립 노출**(프록시를 거치지 않는 직접 접속)로 운영해야 하며, 이 사실을 인스턴스 상세 페이지의 프록시 카드와 생성 화면에 안내 문구로 명시한다(FR-1g와 짝을 이루는 프론트엔드 안내).
- FR-2: 선택한 구동기와 버전 조합에 대해, 앱은 각 구동기의 공식 배포처(예: Mojang `version_manifest.json`(Vanilla), PaperMC `fill` API(Paper/Folia), PurpurMC API, LeafMC API, Jenkins(Pufferfish), Fabric Meta API, NeoForged Maven + 인스톨러 + `ServerStarterJar`, Velocity 공식 배포처)에서 실행 파일을 자동으로 다운로드하고 설치해야 한다.
- FR-3: 사용자는 목록에 없는 구동기를 직접 `.jar` 파일로 업로드하여 서버를 생성할 수 있다.
- FR-4: 기존 서버 인스턴스의 구동기를 다른 구동기/버전으로 교체할 수 있으며, 이 과정에서 월드 데이터는 보존되어야 한다.
- FR-4a: **(결정 반영)** 실제 구현은 **"같은 구동기·같은 마인크래프트 버전 안에서 다른 빌드로 교체"**로 범위를 좁혔다(`handleReinstallLoader`) — 로더 종류나 마인크래프트 버전 자체를 바꾸는 기능은 월드/플러그인 호환성이 깨질 위험이 커서 CraftDeck이 자동화하지 않으며 제공하지 않는다. 배포 API가 버전 하나당 여러 빌드를 노출하는 로더(Paper/Purpur/Folia/Leaf/NeoForge)는 `GET /api/loaders/{loader}/builds`로 조회한 특정 빌드를 지정해 재설치할 수도 있다(`BuildLister`); 지정하지 않으면 기존과 같이 항상 최신 빌드를 받는다. 수동으로 업로드한 커스텀 로더(FR-3)는 재다운로드할 배포처 자체가 없으므로, 새 jar를 파일 탭에서 직접 업로드하는 방식만 지원한다.

### 4.2 플러그인/모드 관리

- FR-5: 플러그인 지원 구동기(Paper/Purpur 등)로 생성된 서버는, **원격 매니페스트 방식**으로 플러그인 목록을 불러와 선택 설치하거나 `.jar` 파일을 직접 업로드하여 적용할 수 있다. 목록 조회 및 다운로드는 **Modrinth API**를 활용하여, 앱을 업데이트하지 않아도 최신 목록이 반영되도록 한다. 1차 출시는 Modrinth만 지원하며, CurseForge는 별도 API 키 신청과 재배포 제약이 있는 라이선스 조건 때문에 1차 범위에서 제외하고 이후 백로그로 미룬다.
- FR-6: 모드 지원 구동기(Fabric/NeoForge 등)로 생성된 서버는, 동일하게 원격 매니페스트(Modrinth API 등)를 통해 모드 목록을 불러와 선택 설치하거나 `.jar` 파일을 직접 업로드하여 적용할 수 있다.
- FR-6a: 원격 매니페스트 조회 시 서버의 구동기 종류/버전과 호환되는 항목만 필터링하여 노출해야 한다 (예: Paper 서버에는 Fabric 전용 모드가 표시되지 않도록).
- FR-6b: 원격 API가 일시적으로 응답하지 않을 경우에도 이미 설치된 플러그인/모드 관리(조회/삭제/활성화)와 직접 업로드 기능은 영향받지 않아야 한다.
- FR-6c: 사용자가 플러그인/모드를 설치할 때 해당 항목이 필수로 요구하는 의존성(예: 특정 모드가 요구하는 API 라이브러리 모드)이 아직 설치되어 있지 않다면, Modrinth API의 의존성 정보를 재귀적으로 조회하여 **필요한 의존성을 함께 자동으로 설치**해야 한다. 사용자가 의존성을 직접 찾아 설치할 필요가 없어야 한다.
- FR-6d: **(SHA-512만 구현)** 원격 매니페스트로 다운로드한 플러그인/모드 파일은 Modrinth API가 제공하는 파일 해시값과 로컬에 받은 파일의 해시를 대조하여 **무결성을 검증**해야 하며, 불일치 시 해당 파일을 설치하지 않고 사용자에게 오류를 안내해야 한다. 실제 구현(`internal/api/handlers_plugin.go`의 `downloadAndVerifySHA512`)은 Modrinth가 함께 제공하는 SHA-1은 쓰지 않고 **SHA-512만** 대조한다 — SHA-512 쪽이 이미 더 강한 해시라 SHA-1까지 이중으로 검증할 실익이 없다고 보고 하나로 좁혔다.
- FR-7: 사용자는 서버에 설치된 플러그인/모드 목록을 조회하고, 개별적으로 활성화/비활성화/삭제할 수 있다.
- FR-8: 플러그인/모드 업로드 시 파일 확장자(`.jar`) 검증과 파일 용량 제한을 적용하여 악성/손상 파일 설치를 방지해야 한다.
- FR-9: 플러그인/모드 설치·삭제·변경 후에는 서버 재시작이 필요함을 UI에서 명확히 안내해야 한다.

### 4.3 서버 인스턴스 운영
- FR-10: 사용자는 동시에 여러 서버 인스턴스를 등록하고 각각 독립적으로 시작/중지/재시작할 수 있다.
- FR-11: **(스왑 연동 반영)** 각 서버 인스턴스는 서버별 CPU/메모리 상한을 설정할 수 있어야 하며(cgroup 기반), 한 서버의 자원 폭주가 다른 서버나 호스트 시스템에 영향을 주지 않아야 한다. 각 인스턴스는 기본적으로 `MemorySwapMax=0`으로 떠서 스왑을 아예 못 쓰도록 격리돼 있다(물리 RAM을 넘기면 느려지는 대신 바로 OOM 종료 — 스왑에 파고든 JVM은 GC 지연으로 사실상 응답 불가 상태가 되는 게 격리 위반에 더 가깝다고 판단). 다만 FR-46의 CraftDeck 자체 스왑파일이 켜져 있으면, 인스턴스 시작 시 이 제한을 풀어(`MemorySwapMax=infinity`) 물리 RAM을 넘는 메모리 할당(`memory_max_mb`)이 실제로 스왑을 통해 동작하도록 한다 — 여러 인스턴스가 하나의 스왑 장치를 나눠 쓰는 건 커널의 스왑 계정 처리에 맡기고, 별도로 인스턴스별 스왑 쿼터를 나누지는 않는다. 웹 UI의 메모리 슬라이더 최대값도 이에 맞춰 "물리 RAM + (스왑 켜짐 시) 스왑 크기 − (Velocity 프록시가 실제로 존재하고 실행 중일 때만) 1GB"로 계산한다(`internal/api/handlers_proxy.go`의 `proxyStatusResponse.Running`, 웹 프론트의 `availableMemoryMB`) — 이전에는 프록시가 꺼져있어도(도메인 미등록으로 아예 삭제된 상태에서도) 항상 1GB를 빼서 독립 인스턴스가 그만큼 손해를 보고 있었다.
- FR-12: 사용자는 서버의 `server.properties` 등 주요 설정 파일을 웹 UI에서 편집할 수 있다.
- FR-12a: **(구현 완료)** FR-12는 자주 쓰는 `server.properties` 키를 라벨/설명과 함께 노출하는 **전용 GUI 폼**(`internal/api/handlers_settings.go`의 `serverPropertyDefs`, 인스턴스 상세 페이지 "게임플레이 설정" 카드)으로 구현했다 — 난이도/게임모드/PVP/최대 인원/화이트리스트 등 약 20개 키를 체크박스·드롭다운·숫자 입력 등 알맞은 컨트롤로 편집하고 한 번에 저장한다. `server-port`/`server-ip`/`online-mode`/`rcon.*` 등 CraftDeck이 직접 관리·의존하는 키는 이 폼에서 아예 제외해(수정 시도 시 400) 게임 포트/RCON 관련 내부 상태와 어긋날 수 없도록 했다.
- FR-12b: **(결정 반영)** GUI 폼의 목적은 일반 사용자가 자주 쓰는 옵션을 안전하게 조정하는 것이고, 목록에 없는 세부 키나 수동으로 구동기를 주입한 인스턴스(FR-3)의 구동기 전용 설정 파일처럼 더 자유로운 편집이 필요한 경우는 **범용 파일 관리자**(`internal/api/handlers_files.go`, 파일 탭 — Windows 파일 탐색기/macOS Finder 스타일의 탐색/더블클릭 편집/드래그 앤 드롭 업로드/다운로드(폴더는 zip)/이름 바꾸기/삭제)를 통해 원본 텍스트를 그대로 편집하도록 역할을 나눴다. 즉 GUI 폼은 "일반 사용자를 위한 큐레이션된 설정", 파일 관리자는 "커스터마이징이 필요한 고급 사용자·커스텀 구동기를 위한 원본 편집" 용도다.
- FR-13: 사용자는 월드 데이터를 백업하고, 백업본으로부터 복원할 수 있다.

### 4.4 콘솔 및 명령어 제어
- FR-14: 사용자는 웹에서 각 서버의 콘솔 로그를 실시간(WebSocket)으로 스트리밍 받아 볼 수 있다. (기본 제공 옵션)
- FR-15: 사용자는 웹 콘솔 입력창을 통해 서버에 자유 형식의 명령어(RCON 경유)를 텍스트로 직접 전송할 수 있다.
- FR-16: 콘솔 로그는 일정 기간/용량 보관 후 로테이션되어야 한다.
- FR-16a: **(구현 완료)** 콘솔 로그(FR-14/15)는 별도 파일이 아니라 systemd 저널(`journalctl -u craftdeck-instance-<id>`)에서 그대로 스트리밍되므로(`internal/api/handlers_console.go`), 로테이션도 journald 자체의 전역 보관 정책으로 구현했다 — 패키지가 `/etc/systemd/journald.conf.d/craftdeck.conf`(`SystemMaxUse=500M`, `MaxRetentionSec=30day`)를 설치하고 `postinst`에서 `systemd-journald`를 재시작해 즉시 적용한다(`packaging/systemd/journald-craftdeck.conf`). journald는 유닛별이 아닌 호스트 전역 보관 한도만 지원하므로, 이 한도는 CraftDeck이 관리하는 모든 인스턴스 유닛의 저널을 합친 총량/기간에 적용된다.
- FR-17: 다음과 같이 실제 운영에서 가장 많이 쓰이는 명령 위주로 GUI 버튼을 제공하여, 명령어 문법을 몰라도 클릭만으로 실행할 수 있어야 한다 (우선순위 순):
  1. 월드 저장 (`save-all`)
  2. 서버 정상 종료 (`stop`)
  3. 플레이어 강퇴 (`kick`)
  4. 플레이어 밴/밴 해제 (`ban`, `pardon`)
  5. 화이트리스트 켜기/끄기/추가/삭제 (`whitelist on/off/add/remove`)
  6. 운영자 권한 부여/해제 (`op`, `deop`)
  7. 전체 공지 방송 (`say`)
  8. 게임모드 변경 (`gamemode`)
  9. 시간 설정 (낮/밤) (`time set day/night`)
  10. 날씨 설정 (맑음/비/뇌우) (`weather clear/rain/thunder`)
  11. 난이도 변경 (`difficulty`)
  12. 접속자 목록 조회 (`list`)
- FR-18: GUI 버튼으로 실행된 명령도 내부적으로 동일한 RCON 경로를 사용하며, 실행 결과가 실시간 콘솔 로그 화면에도 함께 표시되어야 한다.
- FR-19: GUI 버튼 구성은 서버의 구동기 종류(Vanilla/Paper 등 지원 명령 차이)에 따라 다르게 노출될 수 있다.
- FR-20: 실시간 콘솔(텍스트 입력)과 GUI 버튼 조작은 같은 화면에서 함께 제공되는 옵션이며, GUI 버튼이 텍스트 콘솔을 대체하지 않고 사용자는 언제든 자유롭게 원문 명령어를 직접 입력할 수 있어야 한다.

### 4.5 네트워크 / 외부 접속
- FR-21: 사용자는 앱 설정에서 "외부 접속 허용"을 켜고 끌 수 있다.
- FR-22: 외부 접속이 켜지면 앱은 UPnP(IGD) 또는 NAT-PMP를 통해 공유기에 포트 포워딩 규칙을 자동으로 등록/해제할 수 있어야 한다.
- FR-23: 공유기가 UPnP/NAT-PMP를 지원하지 않거나 자동 설정이 실패할 경우, 사용자가 직접 설정해야 할 포트 포워딩 정보(내부 IP, 포트, 프로토콜)를 화면에 안내해야 한다.
- FR-24: 앱은 현재 등록된 포트 포워딩 규칙 목록을 조회하고, 사용자가 개별적으로 삭제할 수 있는 UI를 제공해야 한다.
- FR-25: 마인크래프트 서버 포트(게임 접속용)와 관리 웹 UI 포트에 대해 각각 독립적으로 외부 노출 여부를 선택할 수 있어야 한다.
- FR-25a: **(결정 반영/구현 완료)** 실제로는 웹 UI 포트와 게임 포트를 따로 켜고 끄지 않고, `network_settings.wan_enabled` **토글 하나가 둘 다 함께 관리**한다 — 켜져 있으면 관리 웹 UI 포트와, 그 시점에 실제로 직접 접속 가능한 모든 인스턴스(Velocity 프록시, 그리고 프록시에 등록되지 않은 독립 노출 서버)의 게임 포트가 자동으로 포트포워딩되고, 인스턴스가 시작/종료될 때마다 그 인스턴스의 포트도 함께 자동으로 열리고 닫힌다(`ReconcileGamePorts`, `internal/api/gameports.go`). 프록시 등록 상태가 바뀌거나(FR-1f) 도메인이 등록/해제될 때도 이 게임 포트 매핑을 다시 계산해 반영한다.

### 4.6 도메인 연결 (DDNS)

사용자는 두 가지 도메인 연결 방식 중 하나를 선택할 수 있으며, 방식에 따라 서버 할당 범위가 다르다.

| 구분 | 무료 서브도메인 (DuckDNS, No-IP, **ipTime DDNS** 등) | 사용자 소유 메인 도메인 (예: Cloudflare로 관리) |
|---|---|---|
| 제공 방식 | 제공자가 발급한 고정 호스트명의 A/AAAA 레코드만 갱신 가능 | 사용자가 도메인 전체 DNS Zone API 접근 권한 보유 |
| 할당 가능 서버 수 | **1개** (제공자 특성상 커스텀 SRV 레코드 생성 불가) | **여러 개** (서브도메인을 자유 생성 가능) |
| 접속 방식 | `myserver.duckdns.org` / `myserver.iptime.org`처럼 IP만 대체, 포트는 기본 포트(25565) 고정 | `survival.mydomain.com`처럼 서브도메인별로 서로 다른 내부 포트에 매핑, SRV 레코드로 포트 은닉 |

- FR-26: 사용자는 무료 서브도메인형 DDNS 제공자(DuckDNS, No-IP, ipTime DDNS 등) 또는 본인이 소유한 메인 도메인(DNS 제공자 API 연동) 중 하나를 선택하여 등록할 수 있다.
- FR-26h: **(당시 최소 골격 → 이후 단계에서 완료)** FR-1f(Velocity는 소유 도메인이 있을 때만 동작)를 위해, 실제 DNS API 연동/SRV 자동 갱신/소유권 검증 없이 **"어떤 방식을 등록했는지"만 기록하는 최소 골격**을 가장 먼저 구현했다(`internal/ddns`, `GET/PUT/DELETE /api/domain/settings`) — `kind`(`main_domain`/`free_subdomain`), 제공자 이름, 호스트명을 입력받아 저장하는 것부터 시작했다. 그 위에 얹어야 했던 FR-26a~g의 실제 어댑터 구현, FR-28~31의 SRV/소유권 검증/자동 IP 갱신은 이후 단계에서 모두 완료됐다 — 각 FR 항목의 "(구현 완료)" 표시 참고.
- FR-26a: 무료 DDNS 제공자 지원 구현 순서는 **DuckDNS → ipTime → 기타 제공자** 순으로 진행한다. 단, ipTime은 아래 FR-26e에 따라 다른 제공자와 성격이 다른 방식으로 지원한다.
- FR-26b: **(조사 결과 반영)** ipTime DDNS는 `ddns.iptime.org`를 제3자가 직접 갱신할 수 있는 공개/비공식 API가 존재하지 않으며, 오직 ipTime 공유기 자체(펌웨어)만 갱신할 수 있다(공식 문서, 국내 커뮤니티, OpenWrt ddns-scripts 지원 목록, 관련 GitHub 이슈 등 다각도로 조사하여 확인). 따라서 CraftDeck은 ipTime 호스트명을 **직접 갱신하지 않으며**, 사용자는 공유기 관리 페이지에서 ipTime DDNS 기능을 켜둔 상태를 유지해야 한다.
- FR-26c: 무료 DDNS 제공자는 하나로 한정하지 않고, **가능한 한 널리 쓰이는 모든 무료 DDNS 제공자를 지원하는 것을 목표**로 한다 (예: DuckDNS, No-IP, ipTime, FreeDNS(afraid.org), Dynu, Cloudflare 무료 도메인 활용 등). 실제 능동 갱신이 가능한 제공자(DuckDNS, No-IP 등)는 공통 어댑터 인터페이스(호스트명 갱신 함수)로 추상화하여, 신규 제공자 추가가 기존 코드 변경 없이 어댑터 구현 추가만으로 가능한 구조로 설계해야 한다.
- FR-26d: 각 무료 DDNS 어댑터는 서로 독립적으로 동작해야 하며, 특정 제공자의 API 장애나 변경이 다른 제공자 어댑터나 앱의 나머지 기능에 영향을 주지 않아야 한다.
- FR-26e: ipTime처럼 능동 갱신이 불가능한 제공자는, 사용자가 공유기에서 이미 설정한 `xxx.iptime.org` 호스트명을 CraftDeck에 등록만 하도록 하고, CraftDeck은 이를 갱신 대상이 아닌 **참조/감시 대상**으로 취급한다. UI에는 "이 제공자는 자동 갱신을 지원하지 않으며 공유기 자체 DDNS 기능에 의존합니다"라고 명확히 안내해야 한다.
- FR-26f: **(구현 완료)** CraftDeck은 ipTime처럼 감시 전용으로 등록된 호스트명에 대해 주기적으로 공개 DNS 조회를 수행하여, 그 결과가 현재 공유기 WAN IP와 일치하는지 확인해야 한다. 불일치가 감지되면(공유기 DDNS 기능이 꺼졌거나 실패한 것으로 추정) 사용자에게 알림을 표시해야 한다. `internal/ddns.CheckMismatch` + 20분 주기 리컨사일 루프로 구현했으며, 프론트 "도메인 연결" 카드에 불일치 경고 문구로 표시된다. IPv4만 비교하며(문구상 "WAN IP"가 IPv6까지 요구하는지는 불명확 — FR-30의 AAAA 자동 갱신은 메인 도메인/DuckDNS 쪽만 구현돼 있고 ipTime 감시에는 IPv6 비교가 없음), 필요하면 확장 가능.
- FR-26g: 목록에 없는 신규 무료 DDNS 제공자에 대한 지원 요청은 GitHub 이슈/요청 채널을 통해 받는다. 유지보수자가 요청을 검토하여 기술적으로 어댑터 추가(능동 갱신 가능 여부 포함)가 가능하다고 판단되는 경우 이후 릴리즈에서 추가하는 방식으로 확장한다.
- FR-27: **(결정 반영/구현 불필요로 판단)** 원래 요구사항은 무료 서브도메인이 정확히 1개의 마인크래프트 서버 인스턴스에만 할당되도록 강제하고, 다른 서버에 추가 연결을 시도하면 UI에서 차단하는 것이었다. 그러나 실제 구현 구조상 이 제약은 이미 저절로 충족되어 별도 차단 로직이 필요 없다: (a) `internal/ddns`의 도메인 설정은 애초에 **싱글턴**이라(`Repository.Set`이 항상 기존 행을 지우고 새로 씀) 무료 서브도메인 자체가 시스템 전체에 한 번에 하나만 등록될 수 있어 "여러 서버에 중복 할당"을 시도할 방법이 없고, (b) 무료 서브도메인만 등록된 상태에서는 FR-1f에 의해 Velocity 프록시가 아예 비활성화되고 모든 서버가 독립 노출(서로 다른 `game_port`)로 전환되므로, 같은 호스트명이라도 포트만 다르면 어느 서버로 접속하는지 항상 명확하다(공인 IP 접속과 동일한 방식) — 프록시의 forced-host 라우팅처럼 "어느 서버로 갈지 모호해지는" 문제 자체가 발생하지 않는다. 따라서 이 항목은 추가 구현 없이 완료로 처리한다.
- FR-28: **(구현 완료)** 사용자 소유 메인 도메인을 등록한 경우, 해당 도메인 하위에 여러 개의 서브도메인(예: `survival`, `creative`, `modded`)을 생성하여 각각 다른 서버 인스턴스에 1:1로 할당할 수 있다. 서브도메인별로 다른 서버로 라우팅하는 것 자체(FR-1c의 forced-host)는 이미 구현되어 있었고, 이번에 그 서브도메인의 **실제 Cloudflare A/AAAA 레코드 자동 생성/갱신**이 추가됐다(`internal/dns.UpsertARecord`/`UpsertAAAARecord`, `internal/api/handlers_proxy.go`의 `SyncMainDomainDNS`) — 인스턴스 상세 페이지에서 서브도메인을 지정하면(`setServerSubdomain`) 그 즉시 해당 서브도메인의 A 레코드(및 공인 IPv6가 있는 경우 AAAA 레코드도)가 생성/갱신된다.
- FR-29: **(결정 반영/구현 완료)** 메인 도메인 방식에서는 서브도메인마다 `_minecraft._tcp.<subdomain>` SRV 레코드를 자동 생성/갱신하여, 사용자가 클라이언트에서 포트 번호 없이 서브도메인 주소만으로 접속할 수 있어야 한다. 원래 문구는 "SRV 레코드의 target 포트는 해당 서버 인스턴스에 할당된 포워딩 포트와 일치해야 한다"였지만, 실제로는 forced-host 서브도메인이 전부 하나의 싱글턴 Velocity 프록시(FR-1c)를 거쳐 들어오므로(서버마다 다른 포트가 아니라 프록시 자신의 포트 하나로 전부 모임), target/port는 항상 `<subdomain>`/프록시의 `game_port`로 고정이다. `internal/dns.UpsertSRVRecord` + `SyncMainDomainDNS`로 구현했으며, FR-28의 A 레코드 생성과 같은 시점(서브도메인 지정 시, 그리고 FR-30의 주기 리컨사일)에 함께 갱신된다.
- FR-30: **(구현 완료)** 공유기 WAN IP가 변경되는 경우, 무료 서브도메인은 A/AAAA 레코드를, 메인 도메인은 A/AAAA 레코드 및 관련 SRV 레코드 전체를 자동으로 갱신해야 한다. 무료 서브도메인 쪽은 DuckDNS의 `ipv6` 파라미터(`internal/ddns/duckdns.go`)로, 메인 도메인 쪽은 Cloudflare AAAA 레코드(`internal/dns.UpsertAAAARecord`)로 둘 다 구현했다 — `internal/ddns.Manager`의 20분 주기 리컨사일 루프(`SyncMainDomainDNS`가 `mainDomainSync` 콜백으로 연결됨, `internal/ddns/manager.go`)가 양쪽 다 갱신한다. 공인 IPv6 주소는 `network.FetchPublicIPv6`(api6.ipify.org)로 조회하며, 대부분의 가정용 회선처럼 공인 IPv6가 아예 없는 경우는 에러로 취급하지 않고 AAAA 갱신만 건너뛴다.
- FR-31: **(결정 반영/구현 완료)** 메인 도메인 연동 최초 등록 시 도메인 소유 확인 절차를 거쳐야 하며, API 토큰은 암호화하여 저장해야 한다. 원래 검토했던 "안내된 TXT 레코드를 사용자가 등록 후 검증" 방식 대신, **API 토큰 자체로 자동 검증**하는 방식을 택했다(운영자가 직접 선택, DDNS 자동 검증 논의 때와 동일한 결정) -- Cloudflare에서 "Edit zone DNS" 템플릿으로 해당 도메인 존 하나만 범위를 제한한 토큰을 발급받아 등록하면, `internal/dns.VerifyZoneAccess`(`GET /zones?name=<도메인>`)가 그 토큰으로 실제 그 존에 접근 가능한지 확인한다 -- 범위가 제한된 토큰이 접근 가능하다는 것 자체가 운영자가 그 Cloudflare 계정에서 그 존을 실제로 통제하고 있다는 증거이므로, 별도의 TXT 레코드 등록/확인 왕복이 필요 없다. 검증에 성공하면 토큰은 `internal/secrets`(AES-256-GCM, DuckDNS 토큰과 동일한 방식)로 암호화해 저장하고, FR-28/29/30 레코드 관리 호출에서 재사용할 Cloudflare 존 ID도 함께 캐싱한다(`ddns_configs.zone_id`) — FR-28/29/30의 실제 레코드 생성/자동 갱신도 이 위에 얹어서 이후 단계에서 모두 구현 완료했다.

### 4.7 인증/보안
- FR-32: 웹 UI 접근에는 로그인(계정/비밀번호)이 필요하다.
- FR-33: **(구현 완료)** 관리 웹 UI 포트를 외부(WAN)에 노출하는 순간, 앱은 사용자의 별도 설정 없이 **자동으로** 보안 강화 조치를 적용해야 한다: (a) HTTPS 활성화 — 메인 도메인이 연결되어 있으면 Let's Encrypt 인증서를 자동 발급/갱신하고, 도메인이 없으면 자체 서명 인증서로 폴백, (b) 로그인 실패 임계값/잠금 시간을 LAN 전용 상태보다 더 엄격한 기본값으로 자동 전환, (c) FR-38에 따른 2FA 등록 강제. (a)는 `internal/tlscert`로 구현: 같은 포트에서 TLS ClientHello 여부를 연결마다 스니핑해 플레인 HTTP/TLS를 자동 판별하므로(WAN 토글 값으로 분기하지 않음 — 그 값은 "포트를 인터넷에 노출했는지"를 나타낼 뿐 개별 연결이 실제로 TLS를 쓰는지와는 무관하고, 실기 테스트에서 WAN 토글 값으로 분기했더니 파이4의 NPM이 평문 HTTP로 연결하는 백엔드 경로가 깨지는 게 확인됨), 기존 리버스 프록시(NPM 등) 뒤에 두거나 포트를 직접 노출하거나 상관없이 항상 올바르게 동작한다. SNI가 등록된 메인 도메인과 정확히 일치할 때만 실제 인증서를 시도하고(그 외엔 자체 서명) — 이 체크가 없으면 IP로 접속하는 모든 시도가 실제 Let's Encrypt 발급을 트리거해 rate limit을 낭비하는 게 실기에서 확인됨. (b)는 FR-35와 함께 구현(`s.loginLockoutPolicy`, `internal/api/handlers_auth.go`) — WAN 노출 시 5회/15분, LAN 전용 시 10회/5분 잠금으로 자동 전환. (c)는 FR-38(WAN 켜기 전 2FA 등록 강제)이 이후 단계에서 구현 완료되면서 같이 끝났다.
- FR-33a: **(결정 반영/구현 완료)** Let's Encrypt 인증서는 HTTP-01 챌린지 방식으로 자동 발급하며, 만료 전 자동 갱신되어야 한다. 원래 문구는 HTTP-01이었지만, FR-28~31에서 이미 메인 도메인마다 Cloudflare API 토큰을 검증/암호화 저장해두고 있어서 **DNS-01(Cloudflare) 방식으로 대체**했다 — 포트 80/443을 별도로 열 필요가 전혀 없고(관리 웹 UI가 어떤 포트에서 돌든 무관), 이미 갖고 있는 자격증명을 그대로 재사용한다(`internal/tlscert.Manager`, `github.com/caddyserver/certmagic` + `github.com/libdns/cloudflare`). 인증서는 `<데이터 디렉터리>/certmagic`에 저장되고, certmagic이 자체적으로 만료 전 자동 갱신을 처리한다. **갱신 실패 시 사전 경고 알림도 구현 완료** — certmagic의 `OnEvent` 훅(`cert_failed`/`cert_obtained`)을 연결해(`internal/tlscert.Manager`), 발급/갱신 실패를 `ddns_configs.cert_renewal_error`(마이그레이션 0007)에 기록하고 성공하면 지운다. "도메인 연결" 카드에 무료 서브도메인의 IP 불일치 경고와 같은 자리에 표시되며, 실패 중에도 자체 서명 인증서로 계속 서빙되므로(만료 전 경고일 뿐 접속 자체가 끊기지는 않음) 운영자가 미리 대응(Cloudflare 토큰 재발급 등)할 시간을 번다.
- FR-34: **(구현 완료)** 외부 접속 활성화 시 경고 메시지와 함께 강력한 비밀번호 설정을 유도해야 한다. "외부 접속" 토글을 켜는 순간 바로 적용되지 않고, 먼저 경고 모달(누구나 이 주소로 로그인 시도 가능하니 비밀번호를 확인하라는 안내 + 비밀번호 변경 모달로 바로 이동하는 버튼)을 띄우고, 사용자가 명시적으로 "이해했습니다, 계속"을 눌러야 실제 `setWANEnabled(true)` 호출이 나간다(`web/src/routes/+page.svelte`의 `onWANToggleChange`/`confirmWANEnable`). 끄는 쪽은 경고 없이 바로 적용된다.
- FR-35: **(구현 완료)** 로그인 실패 다수 시 계정 잠금/지연(rate limiting) 처리를 한다 (FR-33의 자동 강화 기본값 적용). `users.failed_attempts`/`locked_until`(마이그레이션 0006)에 계정별로 기록하며, 이미 잠긴 계정은 비밀번호 확인 자체를 건너뛰고 바로 429로 거부한다(`internal/auth.Repository.RecordFailedLogin`/`ResetFailedLogins`, `handleLogin`). 임계값/잠금 시간 자체는 FR-33(b)이 결정한다(WAN 노출 시 5회/15분, LAN 전용 시 10회/5분 — 요구사항에 구체적 수치가 명시돼 있지 않아 임의로 정한 기본값).
- FR-36: 2FA는 오직 **관리 웹 UI 로그인**에만 적용되는 보안 계층이며, 마인크래프트 게임 클라이언트 접속(게임 포트)에는 적용되지 않는다 (마인크래프트 프로토콜 자체가 2차 인증 단계를 지원하지 않으므로 해당 없음).
- FR-37: **(결정 반영/구현 완료)** 관리 웹 UI 포트가 외부(WAN)에 노출된 상태에서는, 관리 웹 UI(설정 페이지 포함) 로그인 시 2단계 인증(TOTP)이 **필수**로 요구되어야 한다. 원래 문구는 "게임 포트만 외부에 노출하고 관리 웹 UI는 LAN 전용으로 유지하는 경우"를 별도 상태로 상정했지만, FR-25a에서 이미 웹 UI 포트와 게임 포트를 하나의 `wan_enabled` 토글로 합치기로 결정했으므로 그 상태 자체가 존재하지 않는다 -- 그래서 실제 조건은 단순히 "이 계정에 2FA가 켜져 있으면(FR-38에 의해 WAN이 켜진 상태에서는 항상 켜져 있음) 로그인 시 매번 요구"로 구현했다(`handleLogin`, `internal/api/handlers_auth.go`). 비밀번호가 맞았는데 2FA 코드를 아직 안 보낸 첫 시도는 실패로 세지 않고 `{"totp_required": true}`를 반환해 프론트가 코드 입력창을 보여주고 재전송하도록 했고, 이후 틀린 코드는 FR-35의 잠금 카운터에 그대로 반영된다.
- FR-38: **(구현 완료)** 사용자가 관리 웹 UI 포트의 외부 노출을 켜려고 시도할 때 계정에 2FA가 아직 등록되어 있지 않다면, 해당 옵션 활성화를 보류하고 2FA 등록(QR 코드 스캔)을 먼저 완료하도록 강제해야 한다. `handleSetNetworkSettings`가 WAN을 켜는 요청에서 `currentUser(r).TOTPEnabled`를 확인해, 아직이면 412로 거부한다(끄는 요청은 항상 허용).
- FR-39: **(구현 완료)** 2FA는 TOTP 표준(RFC 6238)을 따르며, QR 코드를 통한 등록과 백업 코드(복구 코드) 발급 기능을 제공해야 한다. `internal/auth`의 기존 TOTP 헬퍼(`pquerna/otp`)를 `handleTOTPSetup`/`handleTOTPVerify`에 실제로 연결했다 -- setup은 시크릿을 생성해 미확정 상태로 저장하고 QR PNG(`skip2/go-qrcode`, base64 데이터 URI)를 반환하며, verify는 코드 하나를 검증해야만 실제로 2FA를 켜고 그 순간 8개의 백업 코드를 생성해 (bcrypt 해시로 저장하고) 평문으로 딱 한 번만 응답에 담아 돌려준다. 이미 켜진 계정의 재설정(시크릿 교체)은 이번 범위에서 의도적으로 막아뒀다 -- 분실 시 복구 경로는 백업 코드.
- FR-40: **(구현 완료)** 플러그인/모드/구동기 `.jar` 업로드 기능은 인증된 사용자만 접근 가능해야 하며, 업로드된 파일은 실행 전 확장자·크기 검증을 거쳐야 한다. 인증은 이미 두 업로드 라우트(`handleUploadPlugin`/`handleUploadServerJar`) 모두 `requireAuth` 뒤에 있어서 충족돼 있었다. 이번에 실기 점검으로 두 가지를 추가/보강했다: (a) 크기 제한이 `ParseMultipartForm`의 `maxMemory` 인자만으로는 실제로 강제되지 않는다는 걸 확인(초과분이 거부되지 않고 디스크의 임시 파일로 무제한 spill됨) — `http.MaxBytesReader`로 요청 본문 자체를 진짜로 제한하도록 고침. (b) 확장자 검사만으로는 임의 파일의 이름만 `.jar`로 바꿔치기하는 걸 못 막으므로, 실제 파일 내용의 ZIP 시그니처(`PK\x03\x04`)를 확인하는 `requireJarMagicBytes`를 추가. 점검 중 `handleUploadPlugin`이 업로드 파일명을 `filepath.Base` 없이 그대로 경로에 이어 붙이던 실제 경로 순회 취약점도 같이 발견해 고쳤다(예: `../../etc/x.jar` 같은 파일명으로 플러그인 디렉터리 밖에 쓸 수 있었음).

### 4.8 설치 및 종속성 관리
- FR-41: `sudo apt install craftdeck` 실행 시, 데비안 패키지의 `Depends`에 명시된 필수 시스템 패키지가 apt에 의해 자동으로 함께 설치되어야 하며, 사용자가 별도로 `apt install openjdk-...` 등을 수동 실행할 필요가 없어야 한다.
- FR-42: 마인크래프트 서버는 버전에 따라 요구하는 Java 메이저 버전이 다르므로(예: 구버전은 Java 8, 1.17~1.20.4는 Java 17, 1.20.5+는 Java 21 계열), 패키지 설치 시점에 **Java 8, 17, 21 JRE 3종을 모두 설치**한다. 서버 인스턴스는 선택된 마인크래프트 버전에 맞춰 이 중 알맞은 Java 버전을 자동으로 선택해 구동하며, 사용자가 별도로 Java 버전을 설치하거나 선택할 필요가 없다.
- FR-42a: (검증 결과 반영) Raspberry Pi OS의 기반인 Debian 12(Bookworm) 공식 저장소에는 `openjdk-17-jre-headless`만 존재하고 `openjdk-8-jre-headless`, `openjdk-21-jre-headless`는 존재하지 않는다. 따라서 Debian 공식 저장소만으로는 FR-42를 만족할 수 없으며, 앱 패키지는 **Eclipse Adoptium(Temurin) APT 저장소를 `postinst`에서 자동 등록**하고 이를 통해 `temurin-8-jre`, `temurin-17-jre`, `temurin-21-jre`(모두 arm64/armhf 빌드 존재)를 설치하는 방식을 사용한다. Debian 기본 OpenJDK와 Adoptium Temurin을 혼용하지 않고 3개 버전 모두 동일 벤더(Adoptium)로 통일하여 유지보수 일관성을 확보한다.
- FR-42b: 향후 마인크래프트가 Java 21보다 상위 버전을 요구하게 될 경우, 이미 등록된 Adoptium 저장소에서 해당 버전의 `temurin-XX-jre` 패키지를 추가로 설치하는 방식으로 대응한다 (별도의 자체 다운로더 없이 동일 매커니즘 재사용).
- FR-42c: Adoptium 저장소의 GPG 공개 키는 설치 시점에 네트워크로 새로 받아오지 않고, `.deb` 패키지 안에 미리 포함시켜 `signed-by` 옵션으로 참조한다 (설치 시 키 서버 접속 실패라는 추가 실패 지점을 없앰).
- FR-42d: `postinst` 실행 중 Adoptium 저장소 자체에 접속할 수 없어 Java 설치가 실패하는 경우, 설치를 조용히 계속 진행하지 않고 **명확한 에러 메시지와 함께 설치를 중단**해야 한다 (Java 없이 서비스만 반쪽으로 설치되는 상태를 방지).
- FR-42e: **(실기 검증 결과 반영)** 실제 테스트에 사용한 라즈베리파이 4는 Debian 12(Bookworm)가 아니라 **Debian 13(trixie)** 이미지였다. Trixie의 공식 저장소에는 `openjdk-21`/`openjdk-25`는 있지만 `openjdk-8`/`openjdk-17`은 없어서, Bookworm과는 정반대로 다른 버전 조합이 빠져 있었다. 이는 Debian 릴리스마다 기본 제공되는 OpenJDK 버전 조합이 계속 달라진다는 뜻이므로, Debian 공식 패키지와 Adoptium을 섞지 않고 **Java 8/17/21을 전부 Adoptium에서 통일 설치**하기로 한 FR-42a의 결정이 실제로도 유효함이 다시 확인됐다. Adoptium 저장소도 `trixie` 코드네임을 정상적으로 인식해 `temurin-8-jre`/`temurin-17-jre`/`temurin-21-jre` 설치가 그대로 성공했다.
- FR-42f: **(실기 검증 결과 반영)** Adoptium 패키지가 실제로 설치하는 Java 실행 파일 경로는 `/usr/lib/jvm/temurin-<메이저>-jre/bin/java`가 아니라 **아키텍처 접미사가 붙은 `/usr/lib/jvm/temurin-<메이저>-jre-<arch>/bin/java`**(예: `temurin-21-jre-arm64`) 형태였다. 이전에 검증 없이 가정했던 경로 패턴은 실제 설치 결과와 달라 수정이 필요했다.
- FR-42g: 마인크래프트가 버전 `26.1`부터 `1.x[.y]` 형식을 벗어난 **year.release 버전 체계**(예: `26.1`, `26.2`)로 전환했으며, 이 체계의 버전들은 권장 Java 런타임이 **Java 25**이다. 이에 따라 패키지 설치 시점에 설치하는 Adoptium JRE 목록에 `temurin-25-jre`를 추가하고(FR-42 확장), `internal/javaruntime`의 버전→Java 메이저 매핑 로직도 `1.`로 시작하지 않는 버전 문자열은 새 체계로 간주해 Java 25로 매핑하도록 갱신했다.
- FR-43: apt 패키지는 설치 후(`postinst`) 전용 시스템 사용자/그룹(예: `craftdeck`)을 생성하고, systemd 서비스 유닛을 자동 등록·활성화(enable + start)하여 설치 즉시 별도 명령 없이 서비스가 구동되어야 한다.
- FR-43a: **권한 분리**: CraftDeck 패널 데몬 자체는 `systemd-run`/cgroup 제어에 필요한 권한(root 또는 이에 준하는 권한)으로 동작한다. 다만 패널이 생성하는 개별 마인크래프트 서버(및 프록시) 프로세스는 **인스턴스마다 서로 다른 전용 시스템 계정**으로 강등되어 실행되어야 하며, 패널 프로세스 자체의 권한을 상속받지 않아야 한다. 공유 계정 대신 인스턴스별 계정을 쓰는 이유는, 한 인스턴스의 플러그인/모드에서 임의 코드 실행(FR-40 리스크)이 발생하더라도 피해 범위가 해당 서버 프로세스와 그 파일에만 한정되고 다른 인스턴스의 월드/설정 데이터에는 접근할 수 없도록 하기 위함이다.
- FR-43b: **(실기 검증 결과 반영)** 인스턴스별 계정은 systemd의 `DynamicUser=yes`가 아니라, 패널이 인스턴스 생성 시 `useradd --system`으로 직접 만드는 **고정 시스템 계정**(예: `mc-<id 앞자리>`)으로 구현한다. 실제 라즈베리파이에서 `DynamicUser=yes` + `StateDirectory=` 조합을 검증한 결과, 구동기 jar를 인스턴스 생성 시점에 미리 받아두는(즉 서비스 최초 시작 전에 파일을 심어두는) 현재 흐름과 맞지 않아 systemd가 "마이그레이션했다"고 로그를 남기고도 실제로는 작업 디렉터리 진입(CHDIR)에서 권한 오류가 발생했다. 패널이 파일을 내려받은 뒤 해당 디렉터리를 이 고정 계정 소유로 `chown`하는 방식이 안정적으로 동작함을 확인했다.
- FR-43c: **(실기 검증 결과 반영)** 인스턴스 작업 디렉터리로 가는 경로상의 모든 상위 디렉터리는 최소한 순회(실행) 권한을 인스턴스 계정에 열어둬야 한다(예: `chmod 0711`) — 최종 리프 디렉터리만 해당 계정 소유로 바꿔도, 상위 디렉터리 중 하나라도 다른 계정이 통과할 수 없는 권한(예: `0750` root 소유)이면 그 지점에서 CHDIR이 거부된다. 이 때문에 인스턴스 데이터 루트는 사용자 홈 디렉터리처럼 기본적으로 폐쇄적인 권한(`0700`)을 갖는 경로 밑에 두면 안 되며, `/var/lib/craftdeck`처럼 다른 시스템 계정이 순회 가능한 경로에 둬야 한다.
- FR-44: `apt remove craftdeck` 시 서비스는 정상적으로 중지되어야 하며, `apt purge` 시에는 설정 파일까지 정리하되, 사용자의 월드 데이터·백업 파일은 명시적 확인 없이는 삭제되지 않아야 한다.
- FR-45: **(결정 반영/구현 완료)** apt 업그레이드(`apt upgrade`) 시 실행 중인 마인크래프트 서버 인스턴스가 강제 종료 없이 안전하게 처리(그레이스풀 셧다운 후 재시작 또는 유지)되어야 한다. `craftdeck` 패키지 자체의 재시작은 이미 안전하다(`craftdeckd`가 재시작돼도 각 인스턴스의 systemd 유닛은 완전히 독립적이라 안 건드림) — 진짜 위험은 apt가 **다른** 패키지(Java 런타임, glibc, systemd 등)를 업그레이드할 때 `needrestart` 류의 도구나 dpkg 자체가 그 라이브러리를 쓰는 프로세스(=실행 중인 마인크래프트 서버)를 저장 없이 재시작/종료시키는 경우다. `packaging/scripts/apt-hook` + `packaging/apt/99craftdeck`(`/etc/apt/apt.conf.d/99craftdeck`로 설치)로 구현: `DPkg::Pre-Install-Pkgs`(트랜잭션에 포함된 패키지 이름을 실제로 알려주는 유일한 훅)로 이번 apt 작업에 `temurin-*`/`openjdk-*`/`systemd`/`libc6`/`libssl`/`libcrypt` 관련 패키지가 있는지 확인해서, 있을 때만 실행 중인 `craftdeck-instance-*.service`를 (SIGTERM으로 월드 저장 후 정상 종료하도록) 멈추고 어떤 게 켜져 있었는지 `/run/craftdeck`에 기록해뒀다가, `DPkg::Post-Invoke`에서 그것만 다시 켠다. 관련 없는 패키지만 설치/제거하는 일반 `apt install foo`는 건드리지 않도록 필터링했다. **커널 업그레이드 이후 관리자가 수동으로 재부팅하는 경우도 별도 훅 없이 이미 안전함을 확인**했다 — `systemd-run`이 만드는 각 인스턴스 유닛은 일반 `systemctl stop`이든 시스템 전체 재부팅/종료든 상관없이 유닛 자신의 정지 절차(SIGTERM → 마인크래프트 자체 셧다운 훅의 월드 저장 → 정상 종료)를 그대로 따르므로, `/lib/systemd/system-shutdown/` 같은 별도 훅은 필요 없다(그 훅은 이미 모든 유닛이 정지된 *이후*에 실행되어 애초에 시점이 너무 늦다). 실제로 손봐야 했던 건 정지 유예 시간(`TimeoutStopSec`)이 기본값(90초)이라 라즈베리파이의 느린 SD카드 저장 속도로는 충분하지 않아 저장 도중 SIGKILL로 강제 종료(월드 파일 손상 위험)될 수 있었던 부분 — `internal/process/supervisor.go`의 `systemd-run` 호출에 `TimeoutStopSec=300`(5분)을 추가해 저장이 끝날 시간을 넉넉히 주면서도, 진짜로 멈춘 프로세스는 여전히 결국 강제 종료되어 재부팅이 무한정 멈추지 않도록 했다.

### 4.9 가상 메모리 (스왑)

- FR-46: **(결정 반영/구현 완료)** 운영자는 CraftDeck 전용 디스크 기반 스왑파일의 크기를 웹 UI에서 설정/해제할 수 있다. 라즈베리파이 OS가 기본 제공하는 RAM 기반 스왑(`systemd-zram-generator`/"rpi-swap", zram — 실기 확인: dphys-swapfile이 아니라 이쪽이 최신 라즈베리파이 OS의 기본값)과는 완전히 별개로 동작하며, 커널이 항상 실제 RAM과 zram을 먼저 소진한 뒤에야 이 디스크 스왑을 쓴다(별도 설정 없이 커널의 기본 메모리 회수 동작). NVMe/SSD처럼 랜덤 쓰기 성능과 쓰기 수명이 넉넉한 저장장치에서, RAM+zram만으로 부족할 상황을 대비한 추가 여유분 용도다 — 반대로 SD카드 부팅 환경에서는 자동으로 켜지지 않으며, 운영자가 직접 켜고 크기를 정해야 한다. `internal/swap` 패키지가 스왑파일 생성/삭제(`fallocate` 실패 시 `dd` 폴백)·`mkswap`·`swapon`/`swapoff`·`/etc/fstab` 등록(재부팅 후에도 유지)을 담당하며, 요청한 크기가 여유 디스크 공간(1GB 안전 여유분 포함)을 넘으면 거부한다. **SD카드/eMMC 자동 감지 및 기능 자체 비활성화도 구현 완료**: `internal/swap.IsSlowStorage`가 CraftDeck 데이터 디렉터리를 실제로 담고 있는 블록 장치를(`findmnt`) 확인해 `mmcblk*`(SD카드/eMMC)이면 `Info.Supported=false`를 내려주고, 프론트는 이 경우 스왑 카드 자체를 렌더링하지 않는다(껐다 켰다 하는 컨트롤을 비활성화하는 정도가 아니라 아예 안 보이게) — 백엔드 `Set`도 동일한 조건으로 한 번 더 거부해 프론트를 우회한 API 직접 호출도 막는다. NVMe(`nvme*`)/SATA·USB(`sd*`) 등 그 외 장치는 전부 지원 대상으로 취급한다.

## 5. 비기능 요구사항

- NFR-1: 관리 서비스 자체는 라즈베리파이 4(4GB) 환경에서 상시 구동 시 유휴 상태 메모리 사용량이 최소화되어야 한다 (JVM 프로세스들에게 자원을 양보).
- NFR-2: 라즈베리파이 4(arm64/armhf), 라즈베리파이 5(arm64) 양쪽 아키텍처를 지원해야 한다.
- NFR-3: 향후 Pi 5 + NVMe(M.2 HAT) 환경으로 확장 시 서버 개수/자원 상한 설정만으로 스케일업이 가능해야 한다.
- NFR-4: apt 패키지 설치/삭제/업데이트가 표준 `apt install/remove/upgrade` 흐름을 따라야 한다.
- NFR-5: 서비스는 시스템 재부팅 시 자동으로 재시작되어야 한다 (systemd enable).
- NFR-6: 외부 인터넷에 노출되는 컴포넌트는 최소 권한 원칙을 따르며, 공유기 설정 변경 이력을 로그로 남겨야 한다.
- NFR-7: 포트 포워딩 자동 설정 기능은 실패 시에도 앱 전체 동작에 영향을 주지 않아야 한다(격리된 예외 처리).
- NFR-8: 구동기/플러그인/모드 다운로드 및 업로드 파일 처리는 저장 공간 부족 상황을 사전에 감지하고 사용자에게 경고해야 한다.
- NFR-9: 데비안 패키지 자체의 `Depends`는 최소한(`ca-certificates` 등)으로 유지하고, Java 8/17/21 런타임은 `postinst`에서 등록하는 Adoptium APT 저장소를 통해 설치한다(FR-42a 참조). Go 정적 바이너리 + pure-Go SQLite 드라이버(cgo 미사용) 채택으로 `libsqlite3` 등 그 외 불필요한 시스템 라이브러리 종속은 피한다.

```
        ▼

   ├─ 내장 SvelteKit 정적 UI (embed.FS)

   ├─ 플러그인/모드 매니저 (큐레이션 목록 설치 + jar 업로드, 활성/비활성 관리)
   ├─ WebSocket 허브 (콘솔 로그 스트리밍 / 텍스트 명령 / GUI 버튼 명령 결과)
   ├─ 프로세스 매니저 (systemd-run으로 MC 서버 기동/자원 제한)
   ├─ RCON 클라이언트 (텍스트 명령·GUI 버튼 명령 공통 실행 경로)

   ├─ 포트 포워딩 매니저 (goupnp/go-nat-pmp → 공유기)
   ├─ DNS 매니저
   │    ├─ 무료 서브도메인 어댑터 (A/AAAA 갱신, 서버 1개 제한)
   │    └─ 메인 도메인 어댑터 (Cloudflare API, 서브도메인 + SRV 레코드 관리)
   └─ 2FA/TOTP 매니저 (pquerna/otp)
        │
        ▼
[마인크래프트 서버 프로세스 #1..N (JVM, cgroup 격리, 서로 다른 내부 포트, 인스턴스별 고정 시스템 계정(User=/Group=)으로 강등되어 실행)]
```

## 8. 리스크 및 고려사항

- 공유기 제조사/펌웨어에 따라 UPnP/NAT-PMP 지원 여부와 동작이 상이할 수 있음 → 수동 설정 가이드가 필수적으로 병행되어야 함.
- 관리 웹 UI 및 게임 포트를 외부에 노출하는 것은 보안 공격면을 넓히므로, 기본값은 "외부 접속 비활성화"로 두고 사용자가 명시적으로 켜도록 해야 함.
- ISP가 CGNAT을 사용하는 경우 포트 포워딩 자체가 불가능할 수 있음 → 이 경우 안내 문구 필요(예: VPN 릴레이 대안 언급 수준).
- 다중 서버 동시 운영 시 Pi 4(4GB) 환경의 실질적 동시 구동 가능 서버 수에 대한 가이드/경고 필요.
- "가능한 모든 무료 DDNS 제공자 지원"을 목표로 하면 제공자마다 API 스펙, 인증 방식, 요청 주기 제한이 제각각이라 어댑터 개발/유지보수 비용이 커짐 → 공통 인터페이스로 추상화하고, 실제 구현은 사용자 요구가 많은 제공자부터 우선순위를 두어 순차적으로 확장하는 로드맵이 필요.
- ipTime DDNS는 제3자가 능동적으로 갱신할 수 있는 API가 존재하지 않음이 조사로 확인되었으므로(공식 문서/커뮤니티/OpenWrt 지원 목록 어디에도 없음), CraftDeck은 이를 갱신이 아닌 감시(DNS 조회 기반 불일치 알림) 방식으로만 지원한다. 사용자가 공유기의 ipTime DDNS 기능을 끄면 CraftDeck이 이를 대신 갱신해줄 수 없다는 점을 UI에서 명확히 안내해야 함.
- 임의의 `.jar` 파일 업로드(구동기/플러그인/모드)는 원격 코드 실행이 가능한 신뢰 콘텐츠이므로, 인증된 사용자만 업로드 가능해야 한다. FR-43a의 권한 분리(패널=root/승격 권한, 서버 프로세스=강등된 별도 계정)로 1차 방어선을 마련했으나, 패널 데몬 자체가 root로 동작하는 구조이므로 패널의 웹 인터페이스 자체에 취약점이 생기면 파급력이 크다는 점은 남는 리스크임 → 웹 입력 검증과 인증을 특히 엄격히 유지해야 함.
- **(해결됨)** 오리지널 Forge는 설치 과정이 벤더 전용 설치 스크립트를 거쳐야 해서 자동 설치 난이도가 높아 결국 지원 목록에서 제외했다(FR-1e). 후속 프로젝트인 NeoForge는 인스톨러(`--installServer`) + 공식 `ServerStarterJar` 조합으로 동일한 문제를 우회해 구현했다.
- Java 8/17/21 세 버전을 한 번에 설치하면 디스크 사용량이 늘어남(3개 JRE 동시 상주) → Pi의 저장 공간이 넉넉하지 않은 경우(특히 SD카드 부팅) 안내 문구가 필요할 수 있음.
- Adoptium APT 저장소는 Debian 공식 저장소가 아닌 서드파티 저장소이므로, 설치 시점에 GPG 키 등록 및 저장소 가용성에 의존하게 됨 → 저장소 접속 실패 시 설치가 중단될 수 있어 명확한 에러 메시지와 재시도 로직이 필요.
- 향후 마인크래프트가 Java 21보다 상위 버전을 요구하게 되면 이미 등록된 Adoptium 저장소에서 추가 패키지를 설치하면 되므로 대응은 비교적 단순하지만, 그 시점에 Adoptium이 여전히 해당 아키텍처(arm64/armhf) 빌드를 제공하는지는 그때 가서 확인이 필요.
- **(실기 검증 결과 반영)** 인스턴스별 전용 계정으로 서버 프로세스를 격리할 때, 최종 작업 디렉터리만 그 계정 소유로 바꾸는 것으로는 충분하지 않다 — 경로상의 모든 상위 디렉터리가 해당 계정에게 최소한 순회(실행) 권한을 열어줘야 하며, 데이터 루트를 사용자 홈 디렉터리처럼 기본적으로 폐쇄적인 경로(`0700`) 밑에 두면 하위 디렉터리 권한을 아무리 바꿔도 소용없다. `/var/lib/craftdeck`처럼 처음부터 순회 가능한 전용 경로를 데이터 루트로 강제하거나, 설치 시 명확히 문서화해야 함.
